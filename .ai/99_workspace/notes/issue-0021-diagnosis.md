## 1. 전반 평가

#19에서 ADR-0005 본질 메시지 프레임을 정렬한 결과 **메시지 골격은 단단**하다.

- 도입부 "소프트웨어가 만드는 가치" → "AI가 진입장벽을 낮춘다" → "비개발자도 소프트웨어를 만든다" 흐름이 docs/slides 양쪽에 박혀 있음
- 결정 룰(직접 지시 vs 소프트웨어로 만들기)·"왜 계획?" 박스도 ADR-0005가 지정한 3곳에 모두 존재
- 비결정성 메커니즘은 "매번 결과가 다를 수 있음" / "동일한 품질로 반복 보장" 결과형으로 우회 — 정렬 OK

따라서 본 이슈의 보완은 **구조 개편이 아니라 다음 두 축의 미세 다듬기**가 합리적이다.

- **A축. 비개발자 공감 다리** — 추상어 옆에 "당신이 매일 보는 그것" 같은 친근화 한 줄
- **B축. 결과 결과형 강화** — 현재 박스가 "특성: ~함" 명사형으로 끝나는 곳에 "그래서 당신은 어떻게 됨"의 결과 한 줄

## 6. 사용자 추가 의견 (누적)

> 진단 노트 검토 후 사용자가 직접 제시한 보완·추가 요구를 시간순으로 누적 기록한다.

### F-1 (2026-05-03) — Why 도입부 신규: 비개발자 학습 동기 3축 카피 ✅ 확정

> 최초에 4축 예시로 발의 → 사용자가 임팩트·현실성·범위 한정을 강화한 **3축 카피**로 재정의 → 확정.

**확정된 3축 카피**

| # | 축 | 카피 |
|---|------|------|
| 1 | **시간 절약** | "당신의 1시간은 얼마입니까?" — 시간이 가장 소중한 자원 |
| 2 | **직업적 생존** | "AI는 당신을 대체하지 않습니다. AI를 쓰는 사람이 당신을 대체합니다." |
| 3 | **능력의 확장** | "이제 코딩을 몰라도 자신의 업무 자동화가 가능합니다." (대규모 개발이 아닌 **개인 도구** 한정 의도) |

**트랙 분리 정책**: 분리하지 않음 (옵션 D — 공통 골격). 1·2번은 트랙 무관 동일 강도. 3번만 학생·일반인 사례 한 줄 동반 검토 ("자신의 업무" → "자신의 학습·일상" 변형).

**원안 4축 → 확정 3축 변경 의도**

- 원 2번 "기획력 강화" → 폐기. 추상 능력은 AI 없이도 가능 → AI 직결성 약함.
- 원 3·4번 "커리어 경쟁력·지속 경쟁력" → 통합·강화 → "직업적 생존" 단일 카피로 도발 톤 ↑. 학생·일반인도 취준·평가 맥락에서 체감 가능하다는 사용자 판단.
- 원 1번 "업무 생산성" → "시간 절약" 도발 질문형 카피로 임팩트 ↑.
- 원안에 없던 "능력의 확장"을 신설 — ADR-0005 본질 메시지("비개발자도 소프트웨어 만든다")의 직접 표현. "자신의" 한정으로 대규모 개발 오해 회피.

**3축 순서**: 결정 보류 (Task 2 본 작성 단계에서 다시 확인) — 후보 두 가지

- (현재 순서) 시간 → 생존 → 확장: 즉각 효용 → 위기감 → 해결책
- (대안) 확장 → 시간 → 생존: 해결책 제시 → 즉각 효용 → 위기 환기

### F-2 (2026-05-03) — Why 후반부 논리 사다리: "왜 후자(소프트웨어 만들기)를 강조하는가" ✅ 확정

> F-1(개인 동기 3축)과 본 강의 본질 메시지(매번 시키기 vs 소프트웨어 만들기) 사이를 연결하는 **방법론 사다리**가 약하다는 사용자 문제 제기 → "반복성"을 핵심 축으로 사다리를 새로 박는다.

**확정된 사다리 (사다리 A — 반복성 기반)**

```
[1] 공감 진입 — "AI에게 매번 시키는 것도 좋은 출발이다" (1단계 인정 → 청중 위축 방지)
        ↓
[2] 사실 환기 — "그런데 우리 일은 대부분 '한 번'이 아니다 — 반복이 일상"
        예: 매주 보고서, 매번 비슷한 회의록, 매학기 반복 학습
        ↓
[3] 비용 누적 — "반복인데 매번 처음부터 시키면:
       ① 매번 같은 지시 다시 입력 (시간 누적)
       ② 매번 결과가 조금씩 다름 (품질 변동)   ← ADR-0005 결과형 우회 자연 결합
       ③ 매번 검수·수정 (이중 비용)"
        ↓
[4] 수렴 — "반복엔 한 번 만들고 100번 쓰는 게 합리적
       → 본인 일에 맞는 작은 도구를 직접 만든다 (= 본 강의 약속)"
```

**보조 비유 (사다리 D — 식당 vs 요리)** — 4단계 마무리에 한 줄로 결합

- 매번 시키기 = 매번 식당에서 시켜 먹기 (편하지만 누적 비용·통제 ↓)
- 소프트웨어 만들기 = 본인 레시피로 자주 만드는 요리 (처음엔 품 들지만 본인 입맛에 맞고 누적 효율 ↑)
- 본 강의는 "당신만의 레시피 1개"를 가져가는 것

**ADR-0005와의 정합**

- 비결정성 메커니즘은 직접 노출하지 않고 [3]단계 ②번에 결과형으로 자연 흡수 ("매번 결과가 조금씩 다름")
- "한 번 만들면 동일한 품질로 반복 보장"은 [4] 수렴 단계의 결론 라인으로 그대로 유지
- 기존 결정 룰 박스(Who 섹션)는 본 사다리의 요약본 역할 — 위치·메시지 유지

### F-3 (2026-05-03) — Why 섹션 전체 위계 재구성 ✅ 확정

> F-1(개인 동기) + F-2(방법론 사다리)를 기존 구조에 어떻게 녹일지의 위계 결정.

**확정된 3단 위계**

```
[개인 동기 — F-1 새 3축]
  시간 절약 / 직업적 생존 / 능력의 확장
        ↓ (그래서 어떻게?)
[방법론 — F-2 사다리 A + D]
  매번 시키기 vs 소프트웨어로 만들기 (반복성 사다리 + 식당 비유)
        ↓
[본 강의 약속 — ADR-0005]
  비개발자도 본인 일의 작은 소프트웨어를 만든다
```

**기존 W1·W2와의 위계 처리** — 옵션 a 채택 후보 (Task 2에서 최종 확정)

- F-1 새 3축이 도입부 본체로 격상
- 기존 W1("소프트웨어가 만드는 가치")·W2("AI가 바꾸는 것")는 보강 카드로 격하 또는 통합
  - W1은 사회·기술 차원 1슬라이드/카드로 압축 가능 (현재 3축 grid 그대로 유지하되 위계 ↓)
  - W2는 F-2 사다리 [4] 수렴 안으로 흡수 가능 ("코딩 진입장벽 ↓"는 능력 확장 카피와 중복)
- 최종 슬라이드/페이지 분할은 Task 2에서 시안 작성하며 확정

## 7. 사용자 의사결정 요청

확정된 항목은 ✅, 미결은 ❓로 표시.

1. ✅ **이슈 범위**: Tier 1·2 + F-1·F-2·F-3 모두 포함 (사용자 변경 요청 없으면 이대로 진행)
2. ❓ **Tier 3 채택 여부**: 의견 미수신 — 기본값은 미적용, 작성 중 자연스럽게 함께 다듬을 수 있는 항목만 부수적 반영
3. ✅ **F-1 3축 카피 확정** — 단, 3축 순서는 Task 2 본 작성 시 시안으로 결정
4. ✅ **F-2 사다리 A + D 결합 확정**
5. ✅ **F-3 위계 옵션 a 후보 확정** — W1·W2 압축·흡수 시안은 Task 2에서 제시 후 사용자 검토
6. ❓ **추가 보완 페이지** 의견 미수신 — 기본값은 진단 노트 범위
7. ❓ **operation-guide.md 제외 동의 여부** — 기본값은 제외

→ ❓ 항목은 Task 2 시안 단계에서 자연스럽게 재확인. Task 1은 본 노트 갱신과 plan 동기화로 종료.
