# ADR-0002: 콘텐츠 발행 구조 — `docs/`(MkDocs Material) SSoT, `slides/`(Slidev) 파생

- **상태**: Accepted
- **결정일**: 2026-04-25
- **연관 이슈**: [Issue #9](https://github.com/scroogy-dev/ai-onboarding/issues/9)
- **영향 범위**: `docs/`, `slides/`(신규), `mkdocs.yml`, `.github/workflows/`, `AI-CONTEXT.md`의 SSoT 규칙

---

## 배경 (Context)

본 프로젝트는 비개발자 대상 AI 활용 교육 콘텐츠를 운영한다. 현재는 MkDocs Material 기반 정적 사이트(`docs/`)가 유일한 발행 표면이며, AI-CONTEXT.md는 이 디렉토리를 콘텐츠 SSoT로 명시하고 있다.

Issue #9에서 발표용 슬라이드를 Slidev로 마이그레이션하면서 **두 번째 발행 표면**이 추가된다. 이 시점에 두 표면 사이의 관계와 SSoT 정책을 정해두지 않으면, 시간이 지나면서 두 표면이 서로 다른 메시지를 말하기 시작하거나, 동일 콘텐츠를 두 곳에서 비대칭으로 유지하는 비용이 누적된다.

핵심 제약:

1. **표면별 호흡 차이** — 문서는 긴 호흡으로 설명을 풀어가는 매체, 슬라이드는 한 화면 한 메시지로 핵심을 전달하는 매체. 같은 콘텐츠라도 정보 밀도·헤딩 입자·시각 강조 방식이 본질적으로 다르다.
2. **매체별 전용 문법** — MkDocs Material은 admonition·content tabs·grid cards 등의 IA 확장을, Slidev는 layout·click·speaker notes 등의 발표 기능을 가진다. 한쪽에서만 의미를 갖는다.
3. **콘텐츠 운영자가 1인** — 자동 빌드 파이프라인을 도입할 만큼의 규모가 아니며, 도구 유지보수 비용이 콘텐츠 작성 비용보다 커지면 안 된다.
4. **메시지 정합성 요구** — "AI 활용 3단계 모델"(ADR-0001) 같은 핵심 개념은 두 표면에서 변형 없이 동일하게 전달되어야 한다.

---

## 결정 (Decision)

### 1. 콘텐츠 SSoT는 `docs/`를 유지

- AI-CONTEXT.md의 기존 규칙(`docs/`가 콘텐츠 SSoT)을 그대로 유지한다.
- 콘텐츠 변경은 항상 `docs/`에서 시작한다.

### 2. `slides/`는 `docs/`로부터 파생된 발표용 표면

- `slides/`는 별도 디렉토리에서 Slidev로 운영한다.
- `slides/`의 본문은 `docs/`의 핵심을 발표 호흡에 맞게 추리고 재구성한 결과물이다.
- Slidev 전용 문법(layout, click, speaker notes 등)은 `slides/`에서 자유롭게 사용한다.

### 3. 동기화는 단방향 (`docs/` → `slides/`)

- 슬라이드를 먼저 고치고 문서를 뒤따라 맞추는 방향은 금지한다.
- 동기화 시점은 **발표 직전 또는 마일스톤 단위**로 운영한다. 매 doc 변경마다 즉시 슬라이드를 갱신할 의무는 없다.

### 4. 변형 없이 양쪽에서 동일해야 하는 자산은 `src:` import 활용

- Mermaid 다이어그램, 정형 표, 정의문 등은 `docs/` 안에 두고 Slidev의 `src:` 슬라이드 import 또는 이에 준하는 방법으로 끌어다 쓴다.
- 처음부터 공유 디렉토리(`source/`, `shared/` 등)를 만들지 않는다. 실제 중복이 발생한 자산만 추출하는 **후행 추출** 방식을 따른다.

### 5. 메시지 정합성의 닻은 ADR

- "3단계 모델"(ADR-0001) 같은 결정 사항은 두 표면 모두 ADR을 변형 없이 따른다.
- 두 표면이 다르게 표현하는 것이 발견되면 ADR이 정답이다.

## 고려한 대안 (Alternatives considered)

### 대안 A. 순수 md `source/` SSoT + `docs/`·`slides/` 두 파생

- **내용**: `source/`에 platform-agnostic 순수 md를 두고, `docs/`는 Material 확장을 입힌 버전, `slides/`는 슬라이드 호흡으로 재구성한 버전을 각각 사람이 유지.
- **장점**: 본문 narrative가 한 곳에서 통제됨.
- **단점**:
  - 본문 한 줄 변경 시 세 파일을 모두 손대게 됨 (단순 복사 불가, 매번 수동 머지). 두 벌 운영보다 비용 증가.
  - `source/`는 학습자도 청중도 보지 않는 파일이라 시간이 지나면 갱신 누락 → 한물 간 초안 보관소화.
  - 매체별 호흡 차이를 흡수하지 못함 (긴 단락 ≠ 슬라이드 3 bullet).
- **기각 사유**: 의도는 정당하나 비용이 두 벌 운영보다 비싸고, 단일 운영자 환경에서 SSoT 위상을 유지하기 어려움.

### 대안 B. `slides/`를 SSoT로 승격

- **내용**: 슬라이드를 먼저 작성하고 문서를 슬라이드에서 파생.
- **장점**: 발표 자료가 항상 최신 상태로 유지됨.
- **단점**: Slidev 전용 문법(layout, click 등)이 MkDocs로 그대로 흘러들어가면 깨짐. `---`이 슬라이드 구분자로 쓰이는 점도 MkDocs에서는 가로줄로 변환되어 본문이 어색해짐.
- **기각 사유**: 기술 호환성 문제로 MkDocs 측이 손상됨.

### 대안 C. 자동 빌드 파이프라인으로 두 산출물 생성

- **내용**: `source/` 또는 `docs/`를 입력으로 변환기를 두어 `slides/`를 자동 생성.
- **장점**: 동기화 자동화.
- **단점**: 매체별 콘텐츠 호흡 차이를 변환 규칙으로 표현하기 어려움. 결국 변환기 메타데이터가 늘어나면서 콘텐츠 프로젝트가 빌드 툴링 프로젝트로 전환됨.
- **기각 사유**: 1인 운영 콘텐츠 프로젝트의 규모에서 정당화되지 않음. 후일 콘텐츠가 다수가 되거나 다중 표면이 늘어나면 재검토 가치가 있음.

## 결과 (Consequences)

### 긍정적 영향

- **현행 규칙 유지** — AI-CONTEXT.md의 "`docs/`가 SSoT" 규칙이 그대로 살아 있어 추가 학습 비용이 없음.
- **표면별 매체 강점 활용** — Material 확장(admonition·tabs·cards)과 Slidev 기능(layout·animation·notes)을 각자의 자리에서 자유롭게 사용.
- **저비용 운영** — 두 파일 유지에 그치고, 동기화 시점도 마일스톤 단위로 제한하므로 콘텐츠 작성에 집중 가능.
- **점진적 추출 가능성** — 운영하면서 실제 중복이 누적되는 자산만 후행적으로 공유 자산으로 승격.

### 부정적 영향·리스크

- **단방향 동기화 누락 가능성** — `docs/` 변경이 `slides/`에 반영되지 않아 발표 시 시점차 노출 가능.
    - *완화*: 발표 마일스톤 직전 `docs/` 최근 변경 이력 확인을 슬라이드 sync 체크리스트에 포함.
- **두 표면 간 메시지 드리프트** — 시간이 지나면서 표현이 미묘하게 달라질 수 있음.
    - *완화*: ADR-0001 등 핵심 결정의 정의를 양쪽이 동일하게 인용하도록 운영. 의심 시 ADR이 정답.
- **공유 디렉토리 미도입에 따른 자산 중복** — 같은 다이어그램을 두 곳에 두는 일이 처음에는 발생.
    - *완화*: 중복이 5건 이상 누적되면 `shared/` 또는 `source/` 도입을 재검토.

### 후속 조치

- Issue #9 Task 1에서 `slides/` 디렉토리 신설 및 Slidev 초기 세팅.
- Issue #9 진행 중 발견되는 공통 자산(다이어그램·표) 후보를 issue summary에 기록.
- 발표 마일스톤 직전 `docs/` 변경 이력 점검을 slides sync 체크리스트로 정착.
- 추후 콘텐츠가 다수 운영자·다중 표면으로 확장되면 본 ADR을 재검토하고 자동 빌드 파이프라인 도입 여부를 결정 (대안 C 재평가).
